---
title: "Limites de débit"
description: "Ce que la passerelle limite, comment elle s'applique à votre groupe, et comment vous adapter lorsque vous recevez un 429."
---

Pour assurer l'équité des capacités entre les clés et la stabilité des services en amont, la passerelle limite le nombre de requêtes qu'une clé peut envoyer dans une fenêtre glissante. Le mécanisme est basé sur le nombre de requêtes et configuré par [groupe](/fr/guides/user-levels) — il n'existe pas de nombre fixe public. Concevez donc votre client pour lire `429` et revenir en arrière plutôt que de coder en dur un débit.

<Note>
  Utilisateurs Asie-Pacifique : utilisez `https://bearlab.space` à la place de `https://bearlab.ai` dans tous les exemples ci-dessous. Consultez [Disponibilité des services](/fr/guides/availability).
</Note>

## Ce qui est limité

La passerelle limite **les requêtes par fenêtre glissante**, pas les jetons. Deux compteurs indépendants s'appliquent à votre clé, tous deux sur la même fenêtre (configurée en minutes par l'opérateur) :

| Compteur | Ce qu'il compte | Quand il se déclenche |
| --- | --- | --- |
| **Requêtes totales** | Chaque requête que vous envoyez dans la fenêtre, **y compris celles échouées** (mauvais paramètres, erreurs d'authentification, erreurs en amont) | Un flot de tentatives ou de requêtes malformées peut épuiser ce quota même si aucune requête ne réussit |
| **Requêtes réussies** | Seulement les requêtes qui ont retourné `< 400` | C'est le plafond de débit pour le trafic fonctionnant |

Un compteur défini à `0` pour votre groupe signifie que cette dimension est **illimitée**. Parce que le compteur de requêtes totales inclut les échecs, une boucle de retry serrée sur des erreurs peut vous limiter par elle-même — corrigez l'erreur sous-jacente au lieu de continuer à marteler.

<Note>
  La passerelle n'applique pas de budget par jeton par minute (TPM) ou par jeton par jour (TPD). Ce qui limite vos dépenses de jetons est votre [solde et tarification par modèle](/fr/guides/pricing), pas un compteur de débit. Le contrôle des coûts et la limitation de débit sont des systèmes distincts.
</Note>

## Comment cela s'applique à votre groupe

Les limites de débit sont attachées au **groupe** de votre clé, le même groupe qui contrôle [les modèles que vous pouvez appeler](/fr/guides/user-levels). Différents groupes peuvent avoir des compteurs différents ; le groupe `default`, les groupes nommés et `auto` peuvent chacun se résoudre à un plafond différent. Il n'y a pas d'« échelle de niveaux » numérique — le groupe est l'unité.

Parce que la limite est par groupe et configurée par l'opérateur, la réponse faisant autorité à « quelle est ma limite » est empirique : envoyez du trafic et observez `429`. Traitez tout nombre que vous voyez cité ailleurs comme un aperçu qui peut changer à mesure que les canaux sont ajoutés ou retirés.

## Quand vous êtes limité

Lorsqu'un des deux compteurs est épuisé, l'API retourne `429 Too Many Requests` avec un corps d'erreur de style OpenAI. Le `message` indique quelle limite vous avez atteinte et la fenêtre :

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
```

```json
{
  "error": {
    "message": "您已达到请求数限制：N分钟内最多请求M次",
    "type": "new_api_error"
  }
}
```

La variante requêtes totales ajoute que les échecs y comptent — lisez le message pour distinguer les deux :

- **Limite de requêtes réussies** — vous envoyez plus de trafic fonctionnant que votre groupe ne le permet ; ralentissez ou passez à un groupe avec un plafond plus élevé.
- **Limite de requêtes totales** (`...包括失败次数...`) — les échecs sont comptabilisés ; vous relancez probablement une requête cassée. Corrigez la requête, puis relancez-la.

Chaque réponse inclut un en-tête `X-Oneapi-Request-Id` ; citez-le si vous contactez le support. Consultez [Gestion des erreurs](/fr/guides/errors) pour l'enveloppe d'erreur complète.

## Comment réagir

<Steps>
  <Step title="Lisez le message 429">
    Il vous dit si vous avez atteint la limite de requêtes réussies ou de requêtes totales, et sur quelle fenêtre. Le cas requêtes totales signifie généralement que vous relancez des erreurs — corrigez-les d'abord.
  </Step>
  <Step title="Reculer exponentiellement avec bruit">
    Sur `429`, attendez et relancez avec des délais doublés plus du bruit aléatoire afin que les clients parallèles ne se heurtent pas. Un calendrier typique : 1s, 2s, 4s, 8s, plafonné à ~30s, avec un plafond de retry.

    <CodeGroup>

    ```python Python
    import time, random, httpx

    def post_with_backoff(url, headers, json, max_retries=5):
        delay = 1.0
        for attempt in range(max_retries):
            r = httpx.post(url, headers=headers, json=json, timeout=120)
            if r.status_code != 429:
                return r
            time.sleep(min(delay, 30) + random.uniform(0, 0.5))
            delay *= 2
        return r  # surface the final 429 to the caller
    ```

    ```javascript JavaScript
    async function postWithBackoff(url, init, maxRetries = 5) {
      let delay = 1000;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        const res = await fetch(url, init);
        if (res.status !== 429) return res;
        const wait = Math.min(delay, 30000) + Math.random() * 500;
        await new Promise((r) => setTimeout(r, wait));
        delay *= 2;
      }
      return await fetch(url, init); // surface the final 429
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="Limiter la concurrence, pas seulement les retries">
    Limiter les requêtes en attente (un petit sémaphore/pool de workers) prévient les rafales qui déclenchent la fenêtre en premier lieu. C'est plus efficace que de relancer après coup.
  </Step>
  <Step title="Arrêtez de relancer aveuglément les échecs non-429">
    Les erreurs d'authentification (`401`), de permission (`403`) et de paramètre invalide (`400`) comptent toujours vers la limite de requêtes totales. Les relancer gaspille votre budget et peut limiter le débit du trafic autrement sain. Relancez uniquement `429` et les erreurs `5xx` transitoires.
  </Step>
</Steps>

<Tip>
  Si une intégration fonctionnant correctement atteint systématiquement le plafond, la solution est un groupe avec une limite plus élevée (ou illimitée) plutôt qu'un throttling client plus strict. Contactez votre gestionnaire de compte pour revoir votre groupe, ou consultez [Niveaux d'utilisateur](/fr/guides/user-levels) pour comprendre comment les groupes sont attribués.
</Tip>
