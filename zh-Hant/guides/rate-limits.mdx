為保證資源公平使用和服務穩定，平臺對請求頻率與 token 用量設有限制（Rate Limits）。

## 限流的形式

| 形式 | 含義 |
| --- | --- |
| **RPM** | 每分鐘最大請求數 |
| **RPH** | 每小時最大請求數 |
| **RPD** | 每天最大請求數 |
| **TPM** | 每分鐘最大 token 數（輸入 + 輸出） |
| **TPD** | 每天最大 token 數 |

## 與使用者等級的關係

限流額度主要由[使用者等級](/zh-Hant/guides/user-levels)決定：

| 等級 | 獲得方式 | 限流 |
| --- | --- | --- |
| **Tier 1** | 註冊後預設 | 5 RPM |
| **Tier 2** | 任意一次充值後自動升級 | 無限制 |

新賬戶預設 Tier 1（5 RPM），適合輕度測試；**任意金額充值後立即升到 Tier 2，取消速率限制**，無需手動申請。

## TPM 與 max_tokens

請求發出時無法預知輸入輸出總 token，系統會用請求引數 `max_tokens` 估算並據此做 TPM 限流，實際生成後再按真實 token 數校正。建議把 `max_tokens` 設得**接近實際需要**，減少因高估觸發限流的誤差。

## 觸發後的表現

短時間內請求數或 token 用量達到上限時，介面返回 `429 Too Many Requests`，後續請求會被暫時拒絕，冷卻時間過後恢復。

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json

{ "error": { "message": "當前分組上游負載已飽和，請稍後再試", "type": "bearlab_api_error" } }
```

## 最佳化建議

* **指數退避重試**：遇 `429` 等待後重試，等待時間逐次加倍。
* **合併批次請求**：當請求數到頂但 token 仍有餘量時，把多個任務合併進一次請求，提升 token 利用率。
* **需要更高額度**：聯絡業務經理申請單獨配額。
