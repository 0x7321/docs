चाबियों के बीच क्षमता को निष्पक्ष रखने और अपस्ट्रीम को स्थिर रखने के लिए, गेटवे सीमित करता है कि एक चाबी एक रोलिंग विंडो में कितनी अनुरोध भेज सकती है। यह तंत्र अनुरोध-गिनती आधारित है, [समूह](/hi/guides/user-levels) प्रति कॉन्फ़िगर किया गया है — कोई निश्चित प्रकाशित संख्या नहीं है, इसलिए अपने क्लाइंट को `429` पढ़ने और वापस जाने के लिए डिज़ाइन करें, हार्ड-कोडिंग दर के बजाय।

<Note>
  एशिया-प्रशांत उपयोगकर्ता: नीचे दिए गए प्रत्येक उदाहरण में `https://bearlab.ai` की जगह `https://bearlab.space` का उपयोग करें। [सेवा उपलब्धता](/hi/guides/availability) देखें।
</Note>

## क्या सीमित है

गेटवे **रोलिंग विंडो में प्रति अनुरोध** को सीमित करता है, टोकन नहीं। आपकी चाबी पर दो स्वतंत्र काउंटर लागू होते हैं, दोनों एक ही विंडो पर (ऑपरेटर द्वारा मिनट में कॉन्फ़िगर किए गए):

| काउंटर | क्या गिनता है | जब यह ट्रिगर होता है |
| --- | --- | --- |
| **कुल अनुरोध** | विंडो में आप भेजने वाले प्रत्येक अनुरोध, **विफल वाले सहित** (खराब पैराम, प्रमाणीकरण त्रुटियां, अपस्ट्रीम त्रुटियां) | पुनरीक्षण की एक बाढ़ या खराब अनुरोध इस को समाप्त कर सकते हैं, भले ही कुछ भी सफल न हो |
| **सफल अनुरोध** | केवल अनुरोध जो `< 400` लौटे | यह कार्यशील ट्रैफिक के लिए थ्रूपुट सीमा है |

आपके समूह के लिए `0` पर सेट काउंटर का मतलब है कि यह आयाम **असीमित** है। क्योंकि कुल-अनुरोध काउंटर विफलताओं को शामिल करता है, त्रुटियों पर एक तंग पुनरीक्षण लूप आपको अपने आप को दर-सीमित कर सकता है — इसके बजाय अंतर्निहित त्रुटि को ठीक करें।

<Note>
  गेटवे एक टोकन-प्रति-मिनट (TPM) या टोकन-प्रति-दिन (TPD) बजट को लागू नहीं करता है। जो आपके टोकन खर्च को बाधित करता है वह एक दर काउंटर नहीं है, बल्कि आपके [शेष और प्रति-मॉडल मूल्य निर्धारण](/hi/guides/pricing) है। लागत नियंत्रण और दर सीमा अलग-अलग सिस्टम हैं।
</Note>

## यह आपके समूह पर कैसे मैप करता है

दर सीमाएं आपकी चाबी के **समूह** से जुड़ी होती हैं, समान समूह जो [आप किन मॉडलों को कॉल कर सकते हैं](/hi/guides/user-levels) को गेट करता है। विभिन्न समूहों में विभिन्न काउंटर हो सकते हैं; `default` समूह, नामकृत समूह और `auto` प्रत्येक एक अलग टोपी में समाधान कर सकते हैं। कोई संख्यात्मक "स्तर सीढ़ी" नहीं है — समूह इकाई है।

क्योंकि सीमा प्रति समूह है और ऑपरेटर-कॉन्फ़िगर है, "मेरी सीमा क्या है" का आधिकारिक उत्तर अनुभवजन्य है: ट्रैफिक भेजें और `429` के लिए देखें। कहीं और उद्धृत किसी भी संख्या को एक स्नैपशॉट के रूप में मानें जो चैनल जोड़े या सेवानिवृत्त होने पर बदल सकता है।

## जब आप सीमित होते हैं

जब कोई भी काउंटर समाप्त हो जाता है, तो API `429 Too Many Requests` को OpenAI-शैली की त्रुटि निकाय के साथ लौटाता है। `message` नाम देता है कि आप किस सीमा से टकराए और विंडो:

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
```

```json
{
  "error": {
    "message": "您已达到请求数限制：N分钟内最多请求M次",
    "type": "new_api_error"
  }
}
```

कुल-अनुरोध वेरिएंट विफलताओं को गिनते हैं कि यह गिनती करता है — संदेश को पढ़ें दोनों को अलग करने के लिए:

- **सफल-अनुरोध सीमा** — आप अपने समूह की तुलना में अधिक कार्यशील ट्रैफिक धकेल रहे हैं; धीमा करें या अधिक टोपी वाले समूह में जाएं।
- **कुल-अनुरोध सीमा** (`...包括失败次数...`) — विफलताओं को गिना जा रहा है; आप संभवतः एक टूटे हुए अनुरोध को पुनः करने का प्रयास कर रहे हैं। अनुरोध को ठीक करें, फिर पुनः करें।

प्रत्येक प्रतिक्रिया एक `X-Oneapi-Request-Id` हेडर ले जाती है; यदि आप सहायता से संपर्क करते हैं तो इसे उद्धृत करें। संपूर्ण त्रुटि लिफाफे के लिए, [त्रुटि हैंडलिंग](/hi/guides/errors) देखें।

## कैसे प्रतिक्रिया करें

<Steps>
  <Step title="429 संदेश पढ़ें">
    यह आपको बताता है कि आप सफल-अनुरोध या कुल-अनुरोध सीमा से टकराए, और किस विंडो पर। कुल-अनुरोध मामला आमतौर पर मतलब है कि आप त्रुटियों को फिर से करने का प्रयास कर रहे हैं — पहले उन्हें ठीक करें।
  </Step>
  <Step title="jitter के साथ घातांक रूप से वापस जाएं">
    `429` पर, प्रतीक्षा करें और दोगुना देरी के साथ पुनः करें, साथ ही यादृच्छिक jitter इसलिए समवर्ती क्लाइंट फिर से टकराएंगे नहीं। एक विशिष्ट कार्यसूची: 1s, 2s, 4s, 8s, ~30s पर सीमित, पुनः करने की सीमा के साथ।

    <CodeGroup>

    ```python Python
    import time, random, httpx

    def post_with_backoff(url, headers, json, max_retries=5):
        delay = 1.0
        for attempt in range(max_retries):
            r = httpx.post(url, headers=headers, json=json, timeout=120)
            if r.status_code != 429:
                return r
            time.sleep(min(delay, 30) + random.uniform(0, 0.5))
            delay *= 2
        return r  # surface the final 429 to the caller
    ```

    ```javascript JavaScript
    async function postWithBackoff(url, init, maxRetries = 5) {
      let delay = 1000;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        const res = await fetch(url, init);
        if (res.status !== 429) return res;
        const wait = Math.min(delay, 30000) + Math.random() * 500;
        await new Promise((r) => setTimeout(r, wait));
        delay *= 2;
      }
      return await fetch(url, init); // surface the final 429
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="समवर्ती सीमा, केवल पुनः प्रयास नहीं">
    इन-फ्लाइट अनुरोधों (एक छोटी सेमाफोर/कर्मचारी पूल) को सीमित करना फटने वाले को रोकता है जो पहले स्थान में विंडो को ट्रिगर करता है। यह तथ्य के बाद पुनः करने की तुलना में अधिक प्रभावी है।
  </Step>
  <Step title="गैर-429 विफलताओं को अंधे से पुनः करना बंद करें">
    प्रमाणीकरण (`401`), अनुमति (`403`), और खराब-पैराम (`400`) त्रुटियां अभी भी कुल-अनुरोध सीमा की ओर गिनती करती हैं। उन्हें फिर से करने से आपके बजट को नष्ट करता है और अन्यथा स्वस्थ ट्रैफिक की दर सीमा कर सकता है। केवल `429` और अस्थायी `5xx` पुनः करें।
  </Step>
</Steps>

<Tip>
  यदि एक कार्यशील एकीकरण सतत रूप से टोपी से टकराता है, तो सुधार एक अधिक सीमित क्लाइंट-साइड थ्रॉटलिंग के बजाय अधिक (या असीमित) सीमा वाला समूह है। आपके समूह की समीक्षा करने के लिए अपने खाता प्रबंधक से संपर्क करें, या [उपयोगकर्ता स्तर](/hi/guides/user-levels) देखें कि कैसे समूहों को असाइन किया जाता है।
</Tip>
