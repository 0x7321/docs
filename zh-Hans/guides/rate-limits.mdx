为保证资源公平使用和服务稳定，平台对请求频率与 token 用量设有限制（Rate Limits）。

## 限流的形式

| 形式 | 含义 |
| --- | --- |
| **RPM** | 每分钟最大请求数 |
| **RPH** | 每小时最大请求数 |
| **RPD** | 每天最大请求数 |
| **TPM** | 每分钟最大 token 数（输入 + 输出） |
| **TPD** | 每天最大 token 数 |

## 与用户等级的关系

限流额度主要由[用户等级](/zh-Hans/guides/user-levels)决定：

| 等级 | 获得方式 | 限流 |
| --- | --- | --- |
| **Tier 1** | 注册后默认 | 5 RPM |
| **Tier 2** | 任意一次充值后自动升级 | 无限制 |

新账户默认 Tier 1（5 RPM），适合轻度测试；**任意金额充值后立即升到 Tier 2，取消速率限制**，无需手动申请。

## TPM 与 max_tokens

请求发出时无法预知输入输出总 token，系统会用请求参数 `max_tokens` 估算并据此做 TPM 限流，实际生成后再按真实 token 数校正。建议把 `max_tokens` 设得**接近实际需要**，减少因高估触发限流的误差。

## 触发后的表现

短时间内请求数或 token 用量达到上限时，接口返回 `429 Too Many Requests`，后续请求会被暂时拒绝，冷却时间过后恢复。

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json

{ "error": { "message": "当前分组上游负载已饱和，请稍后再试", "type": "bearlab_api_error" } }
```

## 优化建议

* **指数退避重试**：遇 `429` 等待后重试，等待时间逐次加倍。
* **合并批量请求**：当请求数到顶但 token 仍有余量时，把多个任务合并进一次请求，提升 token 利用率。
* **需要更高额度**：联系业务经理申请单独配额。
