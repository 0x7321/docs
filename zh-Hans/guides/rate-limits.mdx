---

为了保持密钥和上游之间的容量公平，网关限制密钥在滚动窗口内可以发送多少请求。该机制是基于请求计数的，按[群组](/zh-Hans/guides/user-levels)配置 — 没有固定的公开数字，因此请设计您的客户端以读取 `429` 并退避，而不是硬编码速率。

<Note>
  亚太地区用户：在下面的每个示例中使用 `https://bearlab.space` 代替 `https://bearlab.ai`。请参阅[服务可用性](/zh-Hans/guides/availability)。
</Note>

## 限制的内容

网关限制**滚动窗口内的请求数**，而不是令牌。两个独立的计数器适用于您的密钥，都在同一个窗口内（由运营商以分钟为单位配置）：

| 计数器 | 计算内容 | 触发时间 |
| --- | --- | --- |
| **总请求数** | 窗口内发送的每个请求，**包括失败的请求**（错误的参数、认证错误、上游错误） | 一阵重试或格式错误的请求可能会用完此配额，即使没有任何请求成功 |
| **成功的请求** | 仅返回 `< 400` 的请求 | 这是正常流量的吞吐量上限 |

为您的群组设置为 `0` 的计数器意味着该维度**无限制**。因为总请求计数器包括失败，在错误上进行紧密重试循环可以自行限制您的速率 — 而不是改为修复基础错误。

<Note>
  网关不强制执行每分钟令牌 (TPM) 或每天令牌 (TPD) 预算。制约您的令牌支出的是您的[余额和每个模型的定价](/zh-Hans/guides/pricing)，而不是速率计数器。成本控制和速率限制是独立的系统。
</Note>

## 如何映射到您的群组

速率限制附加到您密钥的**群组**，这是限制[您可以调用的模型](/zh-Hans/guides/user-levels)的同一个群组。不同的群组可以有不同的计数器；`default` 群组、具名群组和 `auto` 可以各自解析为不同的上限。没有数字"层级阶梯" — 群组是单位。

因为限制是按群组且由运营商配置的，「我的限制是什么」的权威答案是实证的：发送流量并观察 `429`。将您在其他地方看到引用的任何数字视为可能随着频道的新增或停用而改变的快照。

## 当您被限制时

当任一计数器用完时，API 返回 `429 Too Many Requests`，具有 OpenAI 风格的错误主体。`message` 命名您遇到的限制和窗口：

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
```

```json
{
  "error": {
    "message": "您已达到请求数限制：N分钟内最多请求M次",
    "type": "new_api_error"
  }
}
```

总请求变体新增失败计入其中 — 读取消息以区分两者：

- **成功请求限制** — 您推送的正常流量超过群组允许的范围；请放慢速度或移到上限更高的群组。
- **总请求限制** (`...包括失败次数...`) — 失败被计入；您可能在重试一个损坏的请求。修复该请求，然后重试。

每个响应都包含一个 `X-Oneapi-Request-Id` 标头；如果您联系支持，请引用它。有关完整错误信封，请参阅[错误处理](/zh-Hans/guides/errors)。

## 如何回应

<Steps>
  <Step title="读取 429 消息">
    它告诉您是否遇到成功请求或总请求限制，以及在什么窗口内。总请求情况通常意味着您在重试错误 — 首先修复这些问题。
  </Step>
  <Step title="用抖动进行指数退避">
    在 `429` 上，等待并以加倍延迟加随机抖动重试，以便并行客户端不会重新碰撞。典型时间表：1s、2s、4s、8s、上限约为 ~30s，具有重试上限。

    <CodeGroup>

    ```python Python
    import time, random, httpx

    def post_with_backoff(url, headers, json, max_retries=5):
        delay = 1.0
        for attempt in range(max_retries):
            r = httpx.post(url, headers=headers, json=json, timeout=120)
            if r.status_code != 429:
                return r
            time.sleep(min(delay, 30) + random.uniform(0, 0.5))
            delay *= 2
        return r  # surface the final 429 to the caller
    ```

    ```javascript JavaScript
    async function postWithBackoff(url, init, maxRetries = 5) {
      let delay = 1000;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        const res = await fetch(url, init);
        if (res.status !== 429) return res;
        const wait = Math.min(delay, 30000) + Math.random() * 500;
        await new Promise((r) => setTimeout(r, wait));
        delay *= 2;
      }
      return await fetch(url, init); // surface the final 429
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="限制并行性，不只是重试">
    限制在途请求（一个小的信号量/工作者池）可以防止触发窗口的暴增。这比事后重试更有效。
  </Step>
  <Step title="停止盲目重试非 429 失败">
    认证 (`401`)、权限 (`403`) 和错误参数 (`400`) 错误仍计入总请求限制。重试它们会浪费您的预算，并可能限制其他正常流量的速率。仅重试 `429` 和暂时 `5xx`。
  </Step>
</Steps>

<Tip>
  如果一个正常工作的集成始终遇到上限，修复方法是一个具有更高（或无限制）限制的群组，而不是更紧密的客户端侧节流。联系您的账户经理以检查您的群组，或请参阅[用户级别](/zh-Hans/guides/user-levels)以了解群组如何分配。
</Tip>
