Every model on BearLab AI is served by one or more **upstream providers** (vendors). Behind a single model name, the gateway routes your request to a healthy provider node; if one fails or times out, it retries another, transparent to your code. Providers are organized into **routing groups**, and your token decides which group(s) it can draw from.

<Note>
BearLab AI is a unified gateway built on New API. The provider catalog below is generated from the live pricing data for this account, so it reflects exactly which vendors and model families are reachable with your key.
</Note>

## Base URLs

Pick the region closest to you. Both regions expose the same models and the same provider routing.

| Region | OpenAI-compatible base URL | Anthropic base URL |
| --- | --- | --- |
| Americas / EU | `https://bearlab.ai/v1` | `https://bearlab.ai` (messages at `/v1/messages`) |
| Asia-Pacific | `https://bearlab.space/v1` | `https://bearlab.space` (messages at `/v1/messages`) |

Authentication is identical everywhere: send your API key as a Bearer token.

```
Authorization: Bearer sk-...
```

<Note>
Some model families are reachable through more than one provider. For example, the national models (Qwen / GLM / DeepSeek / Kimi / MiniMax) are all served through the **Alibaba Bailian** aggregator rather than separate native vendor routes. The gateway picks an available route for you, so you rarely need to name a specific provider.
</Note>

<Warning>
Group access is bound to your key. Naming a group or provider you do not have permission for in a request will not unlock it — the gateway only routes within what your token is entitled to.
</Warning>

## Configuring routing in a token

When you create or edit a token you set its default routing:

- **Provider / group list**: leave empty (recommended) to route across everything your key can reach; set a single entry to restrict to it; set multiple (comma-separated) to try them in order or by the sort strategy.
- **Provider-only mode**: off (default) prioritizes the listed providers but falls back to others; on strictly limits to them and returns an error if all fail.
- **Sort strategy**: manual (default), latency, price, or throughput.

## Overriding per request

Override your token's defaults for a single request via headers:

| Header | Effect |
| --- | --- |
| `X-Provider-Order` | Providers/groups and order to use for this request |
| `X-Provider-Only` | `"true"` enables strict mode (no fallback) |
| `X-Provider-Sort` | Override sort strategy (`latency` / `price` / `throughput`) |
| `X-Provider-Ignore` | Providers to skip, stacked on the token config |

> Providers and groups requested via headers are still subject to your account permissions.

### Example

<CodeGroup>

```bash Americas / EU (bearlab.ai)
curl https://bearlab.ai/v1/chat/completions \
  -H "Authorization: Bearer sk-..." \
  -H "Content-Type: application/json" \
  -H "X-Provider-Sort: latency" \
  -d '{
    "model": "gpt-5.5",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

```bash Asia-Pacific (bearlab.space)
curl https://bearlab.space/v1/chat/completions \
  -H "Authorization: Bearer sk-..." \
  -H "Content-Type: application/json" \
  -H "X-Provider-Sort: latency" \
  -d '{
    "model": "gpt-5.5",
    "messages": [{"role": "user", "content": "Hello"}]
  }'
```

</CodeGroup>

## Discovering live providers

The list above is a snapshot. To see exactly which models your key can reach right now, query the models endpoint — the gateway auto-detects the protocol from your auth headers:

<CodeGroup>

```bash OpenAI protocol
curl https://bearlab.ai/v1/models \
  -H "Authorization: Bearer sk-..."
```

```bash Anthropic protocol
curl https://bearlab.ai/v1/models \
  -H "x-api-key: sk-..." \
  -H "anthropic-version: 2023-06-01"
```

</CodeGroup>

Each entry includes `id`, `owned_by`, and `supported_endpoint_types`, so you can tell which protocols a given model accepts before you call it. See [Models](/en/guides/models) for the full catalog and [Pricing](/en/guides/pricing) for per-model rates.
