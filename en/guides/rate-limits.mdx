description: "What the gateway limits, how it maps to your group, and how to back off when you hit 429."

To keep capacity fair across keys and upstreams stable, the gateway caps how many requests a key may send in a rolling window. The mechanism is request-count based, configured per [group](/en/guides/user-levels) — there is no fixed published number, so design your client to read `429` and back off rather than hard-coding a rate.

<Note>
  Asia-Pacific users: use `https://bearlab.space` in place of `https://bearlab.ai` in every example below. See [Service Availability](/en/guides/availability).
</Note>

## What is limited

The gateway limits **requests per rolling window**, not tokens. Two independent counters apply to your key, both over the same window (configured in minutes by the operator):

| Counter | What it counts | When it trips |
| --- | --- | --- |
| **Total requests** | Every request you send in the window, **including failed ones** (bad params, auth errors, upstream errors) | A flood of retries or malformed requests can exhaust this even if nothing succeeds |
| **Successful requests** | Only requests that returned `< 400` | This is the throughput ceiling for working traffic |

A counter set to `0` for your group means that dimension is **unlimited**. Because the total-request counter includes failures, a tight retry loop on errors can rate-limit you on its own — fix the underlying error instead of hammering.

<Note>
  The gateway does not enforce a token-per-minute (TPM) or token-per-day (TPD) budget. What constrains your token spend is your [balance and per-model pricing](/en/guides/pricing), not a rate counter. Cost control and rate limiting are separate systems.
</Note>

## How it maps to your group

Rate limits are attached to your key's **group**, the same group that gates [which models you can call](/en/guides/user-levels). Different groups can carry different counters; the `default` group, named groups, and `auto` may each resolve to a different cap. There is no numeric "tier ladder" — the group is the unit.

Because the limit is per group and operator-configured, the authoritative answer to "what is my limit" is empirical: send traffic and watch for `429`. Treat any number you see quoted elsewhere as a snapshot that can change as channels are added or retired.

## When you are limited

When either counter is exhausted, the API returns `429 Too Many Requests` with an OpenAI-style error body. The `message` names which limit you hit and the window:

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
```

```json
{
  "error": {
    "message": "您已达到请求数限制：N分钟内最多请求M次",
    "type": "new_api_error"
  }
}
```

The total-request variant adds that failures count toward it — read the message to tell the two apart:

- **Successful-request limit** — you are pushing more working traffic than your group allows; slow down or move to a group with a higher cap.
- **Total-request limit** (`...包括失败次数...`) — failures are being counted; you are likely retrying a broken request. Fix the request, then retry.

Every response carries an `X-Oneapi-Request-Id` header; quote it if you contact support. See [Error Handling](/en/guides/errors) for the full error envelope.

## How to respond

<Steps>
  <Step title="Read the 429 message">
    It tells you whether you hit the successful-request or total-request limit, and over what window. The total-request case usually means you are retrying errors — fix those first.
  </Step>
  <Step title="Back off exponentially with jitter">
    On `429`, wait and retry with doubling delays plus random jitter so concurrent clients do not re-collide. A typical schedule: 1s, 2s, 4s, 8s, capped at ~30s, with a retry ceiling.

    <CodeGroup>

    ```python Python
    import time, random, httpx

    def post_with_backoff(url, headers, json, max_retries=5):
        delay = 1.0
        for attempt in range(max_retries):
            r = httpx.post(url, headers=headers, json=json, timeout=120)
            if r.status_code != 429:
                return r
            time.sleep(min(delay, 30) + random.uniform(0, 0.5))
            delay *= 2
        return r  # surface the final 429 to the caller
    ```

    ```javascript JavaScript
    async function postWithBackoff(url, init, maxRetries = 5) {
      let delay = 1000;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        const res = await fetch(url, init);
        if (res.status !== 429) return res;
        const wait = Math.min(delay, 30000) + Math.random() * 500;
        await new Promise((r) => setTimeout(r, wait));
        delay *= 2;
      }
      return await fetch(url, init); // surface the final 429
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="Cap concurrency, not just retries">
    Limiting in-flight requests (a small semaphore/worker pool) prevents bursts that trip the window in the first place. This is more effective than retrying after the fact.
  </Step>
  <Step title="Stop retrying non-429 failures blindly">
    Auth (`401`), permission (`403`), and bad-parameter (`400`) errors still count toward the total-request limit. Retrying them wastes your budget and can rate-limit otherwise-healthy traffic. Only retry `429` and transient `5xx`.
  </Step>
</Steps>

<Tip>
  If a working integration consistently hits the cap, the fix is a group with a higher (or unlimited) limit rather than tighter client-side throttling. Contact your account manager to review your group, or see [User Levels](/en/guides/user-levels) for how groups are assigned.
</Tip>
