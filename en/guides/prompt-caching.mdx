---
title: "Prompt Caching"
description: "Use Anthropic and OpenAI prompt caching through the gateway, and keep multi-turn requests on the same backend channel so caches actually get reused."
---

Prompt caching lets a provider reuse the computation for a repeated context prefix, **cutting latency and the cost of input tokens**. BearLab AI passes provider caching through unchanged and adds *channel affinity* so that consecutive requests from the same caller land on the same backend, which is what makes cross-request caching pay off on a multi-channel relay.

<Note>
  BearLab AI serves two regions. US / EU users call `https://bearlab.ai`; Asia-Pacific users call `https://bearlab.space`. Both regions share the same key, model catalog, and caching behavior. See [Service Availability](/en/guides/availability).
</Note>

## How provider caching works

Caching is implemented entirely by the upstream provider; the gateway forwards the request and reports the cache fields back to you. The two protocols differ in whether you have to opt in.

<Tabs>
  <Tab title="Anthropic (opt-in)">
    Claude caching is **explicit**. Mark a cache breakpoint with `cache_control` on a content block (or at the request level). Everything up to and including that block is cached.

    - Marker: `"cache_control": {"type": "ephemeral"}`
    - Default TTL: **5 minutes**, refreshed on every cache hit. A 1-hour TTL is available at 2x base input price.
    - Usage fields in the response: `cache_creation_input_tokens` (written on the first call) and `cache_read_input_tokens` (served from cache afterward).

    Put stable content (system prompt, long documents, tool definitions) before the breakpoint so the cached prefix stays identical across turns.
  </Tab>
  <Tab title="OpenAI (automatic)">
    OpenAI caching is **automatic** — no request changes required, no extra fee.

    - Triggers on prompts of **1024 tokens or more**; shorter prompts are never cached.
    - Usage field in the response: `usage.prompt_tokens_details.cached_tokens` (how many input tokens were a cache hit; `0` when nothing was cached).
    - The optional `prompt_cache_key` field is combined with the prefix hash to **improve routing and hit rate** when many requests share a long common prefix.
  </Tab>
</Tabs>

## Why relays lower the hit rate

A cache is tied to the exact backend account that created it — it **cannot be shared across accounts**. That is a provider-side rule, not a gateway one.

Because BearLab AI load-balances across multiple upstream channels, two consecutive requests from one caller are not guaranteed to hit the same backend. When they land on different channels, the earlier cache is invisible and the prefix is billed at full price.

## Channel affinity keeps you on one channel

To solve this, the gateway runs **channel affinity**: once a request succeeds on a channel, later requests carrying the same affinity value are routed back to that same channel for a TTL window, so the provider-side cache stays reachable.

Affinity is keyed off a value in your request body. The default rules are:

| Protocol | Path | Affinity key (request body field) |
| --- | --- | --- |
| OpenAI-compatible | `/v1/responses` | `prompt_cache_key` |
| Anthropic-compatible | `/v1/messages` | `metadata.user_id` |

Send a **stable value** in that field across the turns of one conversation, and the gateway pins them to one channel. Use a fresh value when you start an unrelated conversation. Affinity entries expire after the rule TTL (1 hour by default) of inactivity.

<Warning>
  Affinity is best-effort. If the pinned channel is unavailable, the request still goes through on another channel — it just won't reuse that channel's cache for that turn. Do not rely on affinity for correctness, only for cost.
</Warning>

  <Tab title="Anthropic (cache_control + affinity)">

      curl https://bearlab.ai/v1/messages \

        -d '{
          "model": "claude-sonnet-4-6",
          "max_tokens": 1024,
          "metadata": {"user_id": "conversation-7f3a"},
          "system": [
            {
              "type": "text",
              "text": "<a long, stable system prompt or document>",
              "cache_control": {"type": "ephemeral"}

  </Tab>

  <Tab title="Anthropic SDK">
    ```python
    import anthropic

    client = anthropic.Anthropic(
        base_url="https://bearlab.ai",   # Asia-Pacific: https://bearlab.space
        api_key="sk-...",
    )

    SYSTEM = "<a long, stable system prompt or document>"
    USER_ID = "conversation-7f3a"  # stable across the whole conversation

    first = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        metadata={"user_id": USER_ID},
        system=[{
            "type": "text",
            "text": SYSTEM,
            "cache_control": {"type": "ephemeral"},
        }],
        messages=[{"role": "user", "content": "Summarize the document."}],
    )

    follow_up = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        metadata={"user_id": USER_ID},  # same value -> same channel -> cache hit
        system=[{
            "type": "text",
            "text": SYSTEM,
            "cache_control": {"type": "ephemeral"},
        }],
        messages=[
            {"role": "user", "content": "Summarize the document."},
            {"role": "assistant", "content": first.content[0].text},
            {"role": "user", "content": "Now list the action items."},
        ],
    )

    print(follow_up.usage.cache_read_input_tokens)
    ```

## Maximizing hits

<Steps>
  <Step title="Order content stable-first">
    Place unchanging content (system prompt, long documents, tool schemas) at the front and put the variable part (the latest user message) last, so the cached prefix stays byte-identical. The longer and more frequently reused the prefix, the more you save.
  </Step>
  <Step title="Set a cache breakpoint (Claude only)">
    Add `cache_control: {"type": "ephemeral"}` after the stable block. OpenAI needs no marker — it caches automatically once the prompt reaches 1024 tokens.
  </Step>
  <Step title="Pin the conversation">
    Send a stable affinity value across turns — `metadata.user_id` for `/v1/messages`, `prompt_cache_key` for `/v1/responses` — and reuse it for the whole conversation. Use a new value for an unrelated conversation.
  </Step>
  <Step title="Verify with usage">
    Check the response usage: `cache_read_input_tokens` (Claude) or `prompt_tokens_details.cached_tokens` (OpenAI). A rising hit count confirms the prefix and the affinity value are both stable.
  </Step>
</Steps>
