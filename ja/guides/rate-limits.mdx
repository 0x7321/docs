キーと上流全体の容量の公平性を保つため、ゲートウェイはキーがローリングウィンドウ内で送信できるリクエスト数を制限しています。このメカニズムはリクエスト数ベースで、[グループ](/ja/guides/user-levels)ごとに設定されます — 固定の公開番号はないため、ハードコードされたレートではなく `429` を読み取ってバックオフするようにクライアントを設計してください。

<Note>
  アジア太平洋地域のユーザーの皆様へ：以下のすべての例で `https://bearlab.ai` の代わりに `https://bearlab.space` をご使用ください。[サービス可用性](/ja/guides/availability)をご参照ください。
</Note>

## 制限の対象

ゲートウェイは**ローリングウィンドウ内のリクエスト数**を制限し、トークン数ではありません。ご使用のキーには2つの独立したカウンターが適用され、どちらも同じウィンドウ内にあります（オペレーターが分単位で設定）：

| カウンター | 計算対象 | トリガー条件 |
| --- | --- | --- |
| **リクエスト総数** | ウィンドウ内で送信するすべてのリクエスト、**失敗したもの含む**（不正なパラメータ、認証エラー、上流エラー） | リトライの連続または不正なリクエストがこのリミットを消費する可能性があり、成功したリクエストがない場合でも発生します |
| **成功したリクエスト** | `< 400` を返したリクエストのみ | これは正常なトラフィックのスループット上限です |

グループに対して `0` に設定されたカウンターは、その側面が**無制限**であることを意味します。リクエスト総数カウンターは失敗を含むため、エラーに対する厳密なリトライループはそれ自体があなたのレートを制限する可能性があります — 代わりに基本的なエラーを修正してください。

<Note>
  ゲートウェイはトークン数/分 (TPM) またはトークン数/日 (TPD) の予算を実装していません。トークン支出を制限するのはレートカウンターではなく、[残高とモデルごとの料金](/ja/guides/pricing)です。コスト管理とレート制限は別のシステムです。
</Note>

## グループへのマッピング

レート制限はご使用のキーの**グループ**に紐付けられており、これは[呼び出し可能なモデル](/ja/guides/user-levels)を制限するのと同じグループです。異なるグループは異なるカウンターを持つ可能性があります。`default` グループ、名前付きグループ、および `auto` はそれぞれ異なるキャップに解析される可能性があります。数値の「階層ラダー」はありません — グループが単位です。

制限はグループごとであり、オペレーターが設定するため、「自分の制限は何か」という信頼できる回答は実験的です：トラフィックを送信して `429` を監視してください。他の場所で見かけた数字は、チャネルが追加または廃止されるにつれて変更される可能性があるスナップショットとして扱ってください。

## 制限される場合

どちらかのカウンターが枯渇すると、APIは `429 Too Many Requests` を返し、OpenAIスタイルのエラーボディが含まれます。`message` はあなたが遭遇した制限とウィンドウを示します：

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
```

```json
{
  "error": {
    "message": "您已达到请求数限制：N分钟内最多请求M次",
    "type": "new_api_error"
  }
}
```

リクエスト総数のバリアントは失敗を含める追加です — メッセージを読んで2つを区別してください：

- **成功したリクエスト制限** — グループが許可するより多くの正常なトラフィックをプッシュしています。速度を落とすか、キャップの高いグループに移動してください。
- **リクエスト総数制限** (`...包括失败次数...`) — 失敗がカウントされています。損壊したリクエストをリトライしている可能性があります。リクエストを修正してからリトライしてください。

すべてのレスポンスには `X-Oneapi-Request-Id` ヘッダーが含まれています。サポートに連絡する場合はそれを引用してください。完全なエラーエンベロープについては、[エラーハンドリング](/ja/guides/errors)をご参照ください。

## 対応方法

<Steps>
  <Step title="429メッセージを読む">
    成功したリクエスト制限またはリクエスト総数制限のいずれに遭遇したか、およびどのウィンドウ内かを示します。リクエスト総数の場合は通常、エラーをリトライしていることを意味します — まずそれらを修正してください。
  </Step>
  <Step title="ジッターを使用した指数バックオフ">
    `429` では、待機してから加倍の遅延とランダムジッターでリトライし、並行クライアントが再度衝突しないようにします。典型的なスケジュール：1s、2s、4s、8s、上限 ~30s、リトライ上限付き。

    <CodeGroup>

    ```python Python
    import time, random, httpx

    def post_with_backoff(url, headers, json, max_retries=5):
        delay = 1.0
        for attempt in range(max_retries):
            r = httpx.post(url, headers=headers, json=json, timeout=120)
            if r.status_code != 429:
                return r
            time.sleep(min(delay, 30) + random.uniform(0, 0.5))
            delay *= 2
        return r  # surface the final 429 to the caller
    ```

    ```javascript JavaScript
    async function postWithBackoff(url, init, maxRetries = 5) {
      let delay = 1000;
      for (let attempt = 0; attempt < maxRetries; attempt++) {
        const res = await fetch(url, init);
        if (res.status !== 429) return res;
        const wait = Math.min(delay, 30000) + Math.random() * 500;
        await new Promise((r) => setTimeout(r, wait));
        delay *= 2;
      }
      return await fetch(url, init); // surface the final 429
    }
    ```

    </CodeGroup>
  </Step>
  <Step title="リトライだけでなく並行性を制限する">
    処理中のリクエストを制限すること（小さなセマフォ/ワーカープール）は、最初の場所でウィンドウをトリガーするバースト現象を防ぎます。これは事後的なリトライよりも効果的です。
  </Step>
  <Step title="429以外の失敗を盲目的にリトライするのをやめる">
    認証 (`401`)、権限 (`403`)、および不正なパラメータ (`400`) エラーは引き続きリクエスト総数制限にカウントされます。それらをリトライすると予算を浪費し、他の正常なトラフィックのレート制限につながる可能性があります。`429` と一時的な `5xx` だけをリトライしてください。
  </Step>
</Steps>

<Tip>
  正常に機能する統合が常にキャップに達する場合、修正方法はより厳密なクライアント側スロットルではなく、より高い（または無制限の）制限を持つグループです。アカウントマネージャーに連絡してグループを確認するか、[ユーザーレベル](/ja/guides/user-levels)を参照して、グループがどのように割り当てられるかを理解してください。
</Tip>
