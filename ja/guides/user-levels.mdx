BearLab AIでは、呼び出すことができる内容は数字の「層」ではなく**モデルグループ**によって決定されます。すべてのAPIキーはグループに属しており、モデルはあなたのグループがそのモデルの許可リストにある場合のみ呼び出し可能です。これは標準的なnew-apiグループメカニズムです：1つのグループ内のキーは、そのグループに割り当てられたチャネルとモデルにのみアクセスできます。

<Note>
  アジア太平洋地域のユーザー：以下のすべての例で `https://bearlab.ai` を `https://bearlab.space` に置き換えてください。[サービス可用性](/ja/guides/availability)を参照してください。
</Note>

## グループがアクセスをゲート化する方法

各モデルには `enable_groups` 許可リストがあります。APIキー(トークン)にはグループがあります。この2つの間の一致がアクセス規則全体です：

| 概念 | 制御する内容 |
| --- | --- |
| **ユーザーグループ** | アカウントに割り当てられたグループ。キーが継承するデフォルトグループ。 |
| **トークングループ** | 個別のAPIキーに設定されたグループ。ユーザーグループと異なる場合があります；`auto` は「優先順位の順に利用可能なグループを選択」を意味します。 |
| **モデル `enable_groups`** | そのモデルを呼び出すことが許可されているグループのリスト。トークングループがその中にない場合、モデルは呼び出し不可能で、`/v1/models` リストに表示されません。 |

[`/v1/models`](/ja/guides/models#fetch-the-live-list) によって返されるカタログは常にトークングループでフィルタリングされるため、ライブリストが「このキーで何を呼び出せるか」の権威ある答えです。

## `auto` グループ

`auto` グループに設定されたトークンは単一のグループに固定されません。プラットフォームは各リクエストに対して優先順位の順に利用可能なグループを選択し、1つのアップストリームが飽和したときにグループ間フェイルオーバーを提供します。復元力が必要で特定のグループを強制する必要がない場合は `auto` を使用してください。既知のモデルセットまたは価格をターゲットにする必要がある場合は、名前付きグループを使用してください。

## このプラットフォームのグループ

これらのドキュメントの生成に使用されたトークンは **`default` グループ** を見ることができます — 各モデルファミリーにまたがる基本混合カタログです([モデルカタログ](/ja/guides/models)を参照)。他のグループはより大きいまたは特殊なモデルセットを公開します。トークン割り当てに利用可能なグループは：

| グループ | 含まれる内容 |
| --- | --- |
| `default` | 基本混合カタログ(最大の単一グループ、すべてのモデルファミリーにまたがる) |
| `openai` | OpenAI / GPTファミリーモデル |
| `grok` | xAI Grokモデル |
| `gemini`, `gemini-guan`, `gemini次数`, `gemini-cli`, `gemini-image` | Geminiルート(標準、公式、リクエストごと、CLI、画像) |
| `claudecode`, `claudecode特价`, `claude次数`, `claude逆向`, `claude官转纯`, `ccmax特价`, `claude max` | Claude/Anthropicルート(Claude Code、割引、リクエストごと、公式など) |
| `claudeAWS`, `claudeAWS2`, `claudeAWS3`, `claudekiro2`, `claudekiro3缓`, `claude反重力` | 代替アップストリーム経由のClaude(AWS Bedrock、Kiro、antigravity) |
| `codex2`, `codex3`, `codext12` | Codex / コード調整ルート |
| `国产` | 国内モデル(DeepSeek、GLM、Qwen、Kimi、MiniMaxなど) |

<Note>
  グループ名とメンバーシップはチャネルが追加または廃止されるに従って変更されます。上記の表をスナップショットとして扱ってください。あなた自身の `/v1/models` レスポンスはキーのグループに限定され、常に真実の情報源です。
</Note>

## グループと価格設定

グループはビリングとアクセスを分離します：同じモデルはグループ間で異なる有効価格を持つことができます。各グループは独自のビリング乗数をモデルベース価格とプロバイダー乗数の上に適用できるためです。グループを選択することは、アクセス決定と価格決定の両方です。

[価格設定とビリング](/ja/guides/pricing)でモデルベース価格と[プロバイダー](/ja/guides/providers)乗数の組み合わせ方を参照し、[レート制限](/ja/guides/rate-limits)でグループごとに適用される制限を確認してください。

## トークンのグループを設定または変更する

APIキーを作成するときに**ダッシュボード、APIトークン**でグループを選択するか、既存のキーのグループを編集します。別のグループのモデルを使用する場合は、トークンをそのモデルの `enable_groups` が含まれているグループに割り当ててください。

<Steps>
  <Step title="APIトークンを開く">
    サインインしてダッシュボード、APIトークンに移動します。
  </Step>
  <Step title="グループを選択">
    トークンの「グループ」フィールドで、名前付きグループまたは優先順位フェイルオーバー用の `auto` を選択します。
  </Step>
  <Step title="/v1/modelsで検証">
    そのキーを使用してライブモデルリストを呼び出し、必要なモデルが存在することを確認してください。

    <Tabs>
      <Tab title="米国 / EU (bearlab.ai)">
        ```bash
        curl https://bearlab.ai/v1/models \
          -H "Authorization: Bearer sk-..."
        ```
      </Tab>
      <Tab title="アジア太平洋 (bearlab.space)">
        ```bash
        curl https://bearlab.space/v1/models \
          -H "Authorization: Bearer sk-..."
        ```
      </Tab>
    </Tabs>
  </Step>
</Steps>

## モデルがグループ外の場合

`enable_groups` がトークングループを含まない `model` を送信する場合、リクエストは拒否されます — キーはそのチャネルに到達できません。トークンをモデルを含むグループに切り替えるか、現在のグループが既に含むモデルを呼び出すことで修正します。

<Tip>
  ダッシュボード、設定で現在のアカウントグループを確認します。ダッシュボード、APIトークンで特定のキーのグループを確認します。
</Tip>
